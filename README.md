# TEM AMCE Executive Dashboard

A single-page dashboard showing how the TEM **AMCE** area (Analytics, Marketing,
Communications, Events) is performing against — and contributing to — TEM's three
areas of focus and the quarterly scorecard.

The defining design decision is to **match the infrastructure to the refresh
cadence.** This data changes quarterly (the formal scorecard) and at most monthly
(operational KPIs) — it is not real-time. So the dashboard is built as a
**static React SPA that reads versioned JSON files from a CDN/object store**, with
the JSON regenerated by a scheduled pipeline. No live database, no app server, no
per-request compute — the cheapest and easiest-to-secure way to serve data at this
cadence.

> **Status:** Prototype built. A React + TypeScript + Vite SPA renders the four
> lenses (Executive Summary, Areas of Focus, Teams, Shared KPIs) from committed
> mock JSON conforming to the [`PRD.md`](PRD.md) data contract, seeded with real
> Q2 2026 values. See **Getting started** below.

## Architecture

```
Source systems ──> Scheduled refresh job (ETL) ──> writes versioned JSON
   (warehouse,        (runs on close /                to object store / CDN
    CRM, etc.)          monthly cron)                       │
                                                            ▼
                                          Static React SPA (CDN-hosted)
                                          fetches JSON at runtime, renders
```

- The SPA compiles to static assets and is served from a CDN/internal static host.
- Data lives in versioned JSON (e.g. `public/data/2026-Q2.json`) plus a small
  `manifest.json` listing available periods. Refreshing numbers never requires a
  code release.
- RAG status is computed once in the pipeline so the deck, dashboard, and any
  export always agree.

See [`PRD.md` §6](PRD.md) for the full data contract (TypeScript types).

## Getting started

```bash
npm install
npm run dev            # local dev server (http://localhost:5173)
npm run build          # static production build to dist/
npm run preview        # preview the production build
npm run validate-data  # validate data files against the contract
```

The four lenses:

- **Executive Summary** — three impact scorecards (one per area of focus), a
  headline-KPI strip, the "what changed" narrative, and the full Q-scorecard
  table grouped by the familiar four categories.
- **Areas of Focus** — each area's lagging outcome as a hero tile, with leading
  indicators grouped by contributing team.
- **Teams** — each of the six leads' owned KPIs and a "contributes to" tag.
- **Shared KPIs** — the de-duplication register: each shared metric once, with
  one accountable owner + contributors (the "report once" rule made visible).

Period switching, "⚠ Unverified" badges (Q2 is transcribed; Q1 is reconciled),
and the shared-once behavior are all wired up.

## Updating the data

Numbers refresh by **editing/adding a JSON file — no code change or redeploy**.
The full playbook (correcting a period, adding a new one, the publish order, and
what the validator enforces) lives in [`public/data/README.md`](public/data/README.md).

```bash
npm run validate-data            # all periods — run before every publish
npm run validate-data 2026-Q2    # one period
```

The validator (`scripts/validate-data.mjs`, zero-dependency) is the guardrail:
it checks required fields/enums, that each `status` matches its `vsGoalPct` (RAG
computed once, never typed twice), that shared KPI ids carry identical values
everywhere (no double-counting), and manifest integrity. It exits non-zero on
error, so wire it into the monthly pipeline's final step and/or CI.
