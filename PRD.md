## 1. Summary

A single-page dashboard that shows how the TEM AMCE area (Analytics, Marketing, Communications, Events) is performing against — and contributing to — the three TEM areas of focus and the existing Q2 2026 scorecard. It mirrors the scorecard executives already read, adds an area-of-focus and per-team breakdown, and enforces the "report shared metrics once" rule so nothing is double-counted.

The defining design decision is to **match the infrastructure to the refresh cadence.** This data changes quarterly (the formal scorecard) and at most monthly (operational KPIs). It is not real-time. So the dashboard is a **static React SPA reading versioned JSON files from a CDN/object store**, with the JSON regenerated by a scheduled pipeline. There is no live database behind the browser, no application server to run, and no per-request compute. That is the most efficient, cheapest, and easiest-to-secure way to serve data at this cadence.

This PRD covers a clickable prototype (mock data) and the production path that reuses the same data contract.

## 4. Refresh cadence → data-serving model (the core decision)

### 4.1 Observed cadence
From the Data Source Inventory and QBR rhythm:

| Layer | Cadence |
|-------|---------|
| Formal AMCE scorecard | **Quarterly** |
| Operational KPIs (campaigns, events, comms) | **Monthly** |
| Source systems (recordkeeping, CRM, marketing automation) | Daily–weekly, but only *consumed* monthly for this view |
| Market share | Quarterly |

**Conclusion:** the fastest the dashboard must change is monthly; the headline scorecard is quarterly. There is no requirement that justifies a live API or always-on database.

### 4.2 Serving model — static JSON over CDN
```
Source systems ──> Scheduled refresh job (ETL) ──> writes versioned JSON
   (warehouse,        (runs on close /                to object store / CDN
    CRM, etc.)          monthly cron)                       │
                                                            ▼
                                          Static React SPA (CDN-hosted)
                                          fetches JSON at runtime, renders
```

- The SPA is **compiled to static assets** (HTML/JS/CSS) and served from a CDN or internal static host. No server runtime.
- Data lives in **versioned static JSON** (e.g., `data/2026-Q2.json`) plus a small `manifest.json` listing available periods and the current one.
- A **scheduled pipeline** (monthly cron, or on-demand at period close) pulls from sources, computes KPI values + RAG, and writes the JSON. Publishing data is decoupled from deploying the app — refreshing numbers never requires a code release.
- **Runtime fetch, not build-time embed:** the app fetches `manifest.json` then the selected period's JSON. This lets non-engineers refresh data by replacing a file, and lets users switch historical periods without a rebuild.

### 4.3 Why this is the efficient, production-ready choice
- **Cost/ops:** static hosting + a monthly job ≈ near-zero running cost; nothing to keep online 24/7, nothing to patch.
- **Performance:** JSON for one quarter is a few KB–tens of KB; long-cache + CDN = sub-second loads; no query latency.
- **Security/governance (key at Fidelity):** no live DB exposed to the browser, minimal attack surface, simple access story (static host behind SSO); aggregate-only data avoids PII handling in the client.
- **Caching/versioning:** immutable per-period files with content hashing; `manifest.json` is the only short-TTL file, used for cache-busting.
- **Reproducibility:** every period is an archived JSON artifact — instant historical views and an audit trail.

> Rule of thumb encoded here: **infrastructure should match cadence.** Real-time infra for quarterly data is wasted cost and added risk.

## 5. Information architecture

1. **Executive Summary** — mirrors the Q2 2026 scorecard.
   - Three impact scorecards: Growth & Retention (Market Share & Retention), Participant Engagement (Engagement & LTV), Scale & Resiliency (Operating Efficiency).
   - Scorecard table by the four familiar categories (Grow TEM's Business, Client Retention, Deepen Participant Engagement, Execute on Key Initiatives) with Actual, Target, vs Goal, vs YoY, QoQ trend, RAG.
   - Headline-KPI strip + a short "what changed" narrative slot.
2. **Area-of-focus pages (×3)** — each area's KPIs, the lagging outcome as hero tile, contributing leading indicators grouped by team, segment cuts (HC/HE; women, nurses, faculty/physicians).
3. **Team pages (×6)** — Analytics, B2C+B2B Marketing (Beth), Comms–Consultants (Eric), Comms–Sales/SCC (Jon), Comms–Implementation (Jason), Events (Kara). Each shows owned KPIs, targets, trend, and a "contributes to" tag.
4. **Shared AMCE KPIs view** — the de-dup register: each shared metric once, with accountable owner + contributors, so the report-once rule is visible.

**Global controls:** period selector (from `manifest.json`), area filter, team filter, segment filter. Persist selections in `localStorage`.

## 6. Data model (the contract)

One JSON document per period. The same schema powers the prototype (mock) and production (pipeline output).

### 6.1 Types (TypeScript)
```ts
type RAG = "green" | "amber" | "red" | "na";
type Area = "growth_retention" | "participant_engagement" | "scale_resiliency";
type LeadLag = "leading" | "lagging";
type Ownership = "owned" | "led_partner" | "shared" | "partner_owned";

interface KPI {
  id: string;                 // "TEM-01"
  name: string;               // "HC/HE Market Share"
  area: Area;
  scorecardLine?: string;     // "Grow TEM's Business"
  ownerTeam: string;          // "Analytics" | "B2C Marketing" | ...
  ownership: Ownership;
  leadLag: LeadLag;
  tier: "core" | "strategic" | "pilot";
  unit: "percent" | "currency" | "count" | "ratio" | "score" | "status";
  value: number | string;
  target: number | string;
  vsGoalPct?: number;         // 100 = on target
  vsYoYPct?: number;
  status: RAG;                // precomputed in the pipeline
  priorPeriodValue?: number | string;
  qoqTrend?: "up" | "down" | "flat";
  priorYTD?: number | string;
  isShared?: boolean;
  contributors?: string[];    // teams contributing to a shared KPI
  source?: string;            // system of record
  refreshCadence?: "monthly" | "quarterly";
  verified?: boolean;         // false until reconciled to source
  note?: string;
}

interface PeriodDocument {
  period: string;             // "2026-Q2"
  generatedAt: string;        // ISO timestamp
  areas: { id: Area; label: string; impactMeasure: string }[];
  scorecard: KPI[];           // executive scorecard lines
  teams: { id: string; label: string; lead: string; kpis: KPI[] }[];
  sharedKPIs: KPI[];          // the de-dup register
  narrative?: string;         // "what changed this period"
}

interface Manifest {
  current: string;            // "2026-Q2"
  periods: string[];          // ["2026-Q1","2026-Q2", ...]
  schemaVersion: string;      // "1.0"
}
```

### 6.2 RAG logic (centralized, computed in the pipeline)
- `vsGoalPct ≥ 95` → **green**; `90–95` → **amber**; `< 90` → **red**; missing → **na**.
- Status is computed once in the pipeline (not in the UI) so the deck, dashboard, and any export agree.

## 7. Tech stack

| Concern | Choice | Rationale |
|--------|--------|-----------|
| Framework | **React + TypeScript via Vite** | Fast static builds; types enforce the data contract |
| Styling | Tailwind CSS (or CSS modules) | Quick, consistent; Fidelity-ish theme tokens |
| Charts | Recharts (or visx for control) | Lightweight; enough for trend lines, bullet/RAG tiles |
| Data access | `fetch` of static JSON + small typed loader | No client DB; no SDKs |
| State | React state + `localStorage` for filters | No global store needed at this size |
| Hosting | Static host / CDN behind SSO | No server runtime; trivial security posture |
| Pipeline | Scheduled job (Python/Node) → JSON to store | Decouples data refresh from app deploys |

GenAI dev-assist is well-suited to scaffold the SPA and the data loader from this contract.

## 8. Non-functional requirements

- **Performance:** first meaningful paint < 1.5s on internal network; period switch < 300ms (data is local/CDN-cached).
- **Availability:** inherits CDN/static-host SLA; no app server to fail.
- **Security/compliance:** SSO-gated; aggregate KPIs only, no PII in client or JSON; security review limited to a static site + a batch job. Data classification confirmed before any internal value lands in a file.
- **Accessibility:** WCAG AA — RAG never encoded by color alone (use icon/label + color); keyboard navigable; legible at exec-projector sizes.
- **Maintainability:** adding/renaming a KPI = a JSON change, not a code change, wherever possible.
- **Auditability:** every published period archived as an immutable JSON artifact.

## 9. Prototype scope vs. production

### Prototype (this build)
- Full IA and components, wired to **committed mock JSON** matching the schema (seeded with real Q2 2026 values).
- Period selector with one or two mock periods to prove time-switching.
- "Unverified" badges and the shared-KPI report-once behavior visible.
- Deployable as static files for clickthrough review with leadership.

### Production (phase 2)
- Replace mock JSON with pipeline output; same schema, no UI rewrite.
- Stand up the scheduled refresh job against approved sources; add reconciliation/verification step that flips `verified: true`.
- SSO + access list; archival of period artifacts.
